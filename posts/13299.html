SQL事务

1.什么是事务

一个数据库的一组操作,必须保证4个特征(ACID):

  • Atomicitry:原子性,一个事务必须被视为一个而不可分割的最小工作单元.整个事务中的所有操作要么全部提交成功,要么全部失败回滚.对于一个事务,不可能只是执行一部分.
  • Consistency :一致性,数据库总是从一个一致性状态转换到另外一个一致性状态.
  • Isolation:隔离性,通常来说,一个事务所有的修改在最终提交以前,对其他事务是不可见的.
  • Durability:持久性,一旦一个事务提交,则其所做的修改就会永远保存在数据库中,即使系统崩溃,修改的数据也不会丢失(持久性的安全性和刷新日志级别有关系,不同的级别对应不同的数据安全级别).

以银行账户转账为例:

-- 新建一个银行储户表 
create table checking(
    customer_id int auto_increment not null,
    balance float not null,
    primary key (customer_id)
)engine=Innodb default charset utf8;

create table saving(
    customer_id int auto_increment not null,
    balance float not null,
    primary key (customer_id)
)engine=Innodb default charset utf8;

insert into checking values(1, 300.00);
insert into saving values(1, 100.00);

-- 开始事务
start transaction;
select * from checking where customer_id=1;
update checking set balance=balance-100.00 where customer_id=1;
update saving set balance=balance+100.00 where customer_id=1;
commit;
-- 事务结束
select * from saving;
select * from checking;

原子性:customer_id=1 的用户它的转账的钱要么完全提交(表checking 减少100,表saving增加100),要么回滚(两个表都不改变)

一致性:体现在数据100 不会因为系统处理到表checking 减少100突然崩溃而消失不见,因为事务还没有提交

隔离性:A操作数据库减少金额,在事务没有提交之前,B操作数据库,看不到表checking 的改变.

持久性:这个操作会被记录到系统日志中

注意:mysql 的Innodb 引擎才支持事务,MyISM 是不支持事务的 .

2.隔离性和隔离级别

当并发事务的时候,可能会出现更新丢失``(lost update),脏读(dirty reads),不可重复读(non-repeatable reads),幻读(phantom reads`) 的问题.

lost update:当两个或多个事务选择统一行数据进行更新时,由于每个事务都不知道其他事务的存在,就会发生丢失更新-最后的更新覆盖了其他事务所做的更新.

dirty reads:一个事务正在对一条记录做修改,在这个事务完成并提交前,这条数据就处于不一致状态.这个时候,另外一个事务也来读取同一条记录,如果不加以控制,第二条事务就读取了这些脏 数据,并据此做了进一步的处理,就会产生未提交的数据依赖关系.这种现象称为dirty reads

non-repeatable reads :一个事务在读取某些数据后的某个时间,再次读取以前读过的数据,却发现其读出的数据已经发生了改变,这种现象叫non-repeatable reads

phantom reads: 一个事务按相同的查询条件重新读取以前检索过的数据,却发现其他事务插入了满足其查询条件的新数据,这种现象叫phantom reads

幻读和不可重复读的区别:不可重复读的重点是修改,在同一事务中,同样的条件,第一次读的数据和第二次读的数据不一样;幻读的重点在于新增和删除,在同一事务中,同样的条件,第一次和第二次的记录数一样.

为了解决上面的问题,数据库引入了隔离级别,每一种级别都规定了一个事务中所做的修改.哪些在事务内可见,哪些在事务外可见.低级别的隔离一般支持更高的并发处理,拥有更低的系统开销.

SQL 的事务级别包括:read uncommitted,read committed,repeateable read ,serializable4种级别一次升高.

read uncommited:读未提交

  • 所有的事务都可以看到其他未提交事务的执行结果
  • 这个级别会引发dirtry reads ,使得其他事务读取到未提交的数据
  • 在实际中很少应用,因为性能不比其他隔离级别好,却缺少其他隔离级别的好处

read committed:读已提交

  • 这是大多数数据库默认的隔离级别(不是Mysql的默认级别)
  • 它满足了隔离的简单定义:一个事务只能看到已经提交的事务结果
  • 会产生问题non-repeateable read .这意味者,同一个事务的select 语句看到结果可能不一样.

repeatable read:可重复读

  • 这是mysql 的默认事务隔离级别
  • 它确保同一事务的多个实例在并发读取数据时,可以看到同样的结果
  • 可能出现的问题phantom read ,幻读是指当某个事务在读取某个范围内的记录时,另外一个事务又在该范围内插入了新的记录,之前的事务再次读取该范围的记录时,会产生幻行
  • InnoDB 和Falcon 存储引擎通过多版本并发控制(MVCC )机制来解决幻读问题;InnoDB 还通过间隙锁来解决幻读问题

serializable:可串行化

  • 这是最高的隔离级别
  • 它通过强制事务排序,使之不可能相互冲突,从而解决幻读问题,简单来说,对于同一行记录,read 会加read lock ,write 会加write lock ,当出现读写锁冲突的时候,后访问的事务必须等前一个事务执行完成,才能执行.
  • 会产生大量的超时现象和锁竞争.

按照时间顺序在不同的隔离级别下,会显示不同的内容

  • 读未提交:事务2的id 在time1 值为2 ,虽然事务1没有提交,但是级别定义的原因,time2\time3值都是2
  • 读已提交:事务2的id 在time1 值为1,一个事务只能看到已经提交的结果time2\time3 值为2
  • 可重复读:事务2的id 在time1 值为1,time2 的值也是1,因为事务在执行期间看到的结果必须是一直的,time3 是2
  • 串行化:事务1在执行id=id+1 时就会被锁住,直到事务2提交后,事务1才会执行,所以time1\time2 的值为1,time3的值为2

3.事务隔离的实现

在mysql中,可重复事务隔离的实现

1.事务控制语句

在mysql 命令行的默认设置下,事务都是自动提交的,也就是执行完一行SQL 语句后,就会马上执行commit 语句.

禁止当前会话的自动提交

  • set autocommit=0: 这个命令会将这个线程的自动提交关掉,意味着执行的SQL 命令在最后关闭连接的时候需要commit 提交才生效.

可以显式的开启一个事务:

  • begin/start transaction :显式的开启一个事务,但不是一个事务的起点,起点是执行的第一条SQL 语句.这种启动方式,一致性视图是(快照)在执行的第一个SQL 语句的时候创建的.

  • start transaction with consistent snapshot :显式的开启一个事务,一致性视图(快照)立刻创建.

提交事务

  • commit:对数据库做的修改称为永久性的.

回滚事务:

  • rollback :回滚会结束用户的事务,并撤销正在进行的所有未提交的修改.

保存点(子事务):

  • savepoint savepoint_name :创建一个保存点,一个事务可以有多个保存点.
  • release savepoint savepoint_name :删除一个事务的保存点,没有指定名字的时候,会报错
  • rollback to savepoint_name:将事务回滚到标记点.

设置事务的隔离级别:

  • set transcation 级别名称

4.可重复读的原理

在Mysql 中有两个view 视图的概念

  • view :虚拟表,可以创建,可以查询.
  • InnoDB 在实现数据库多版本并发控制(MVCC ) 用到的一致性读视图(consistent read view),用于支持read commited,repeatable read 隔离级别的.它没有物理结构,作用是在事务执行期间定义事务能看到的数据.

1.MVCC

MVCC(Multiversion Concurrency Control):多版本并发控制,用来解决读写锁造成的多个长时间的读操作饿死写操作的问题.每个事务读到的数据项都是一个历史快照(snapshot)并依赖实现的隔离级别,写操作不覆盖已经有的数据,而是创建一个新的版本,直到所有在操作提交的时候才变成可见.

2. 锁的展示:

  • 读锁/共享锁/S Lock :若事务T 对数据对象A加上S Lock ,则事务T 可以读A但是不能修改A.其他事务只能再对A加S Lock,而不能加X Lock ,直到事务T 释放了A上的S Lock .这保证了其他事务可以读A,但是在事务T 不释放锁之前不能修改A
  • 写锁/排他锁/X Lock:若事务T 对数据对象A加上X Lock ,则事务T 可以读A也可以修改A.其他事务不能再对A添加任何锁.直到事务T 释放了A上的锁.这保证了其他事务在锁没释放之前不能做任何操作.
  • 表锁:操作对象的是数据表.事务T 对整个表加S Lock ,其他事务可读不可写.如果是X Lock,其他事务增删改都不行.系统的开销小,但可并发能力低.
  • 行锁:操作对象是数据表中的一行.MVCC 使用比较多的技术.MyISM 不能使用.整个技术的实现是源自存储引擎,不是mysql 服务器.系统的开销大,可并发能力高.

3.意向锁

InnoDB 存储引擎支持多粒度锁定,这种锁定允许事务的行锁和表锁同时存在.为了支持在不同粒度上进行加锁操作,InnoDb存储引擎引入了一种称为意向锁的锁方式,意向锁是将锁定的对象分为了多个层次,意味者事务在更细粒度上的加锁.

意向锁是表级别的锁(和表锁不一样).设计的目的主要是为了在一个事务中揭示下一行被请求的锁类型.

  • IS Lock :意向共享锁,事务想要获取一张表中某几行的共享锁

  • IX Lock:意向排它锁,事务想要获取一张表中某几行的排他锁

  • 对一张表进行锁操作的时候,如果没有意向锁,需要遍历整个表,看是否有行锁的存在,而有意向锁后,只要比较兼容性,就可以得到结果.

  • 意向锁不会和行锁发生冲突,它允许行锁和表锁的同时存在.但是会和表锁发生冲突,这也很好理解,意向读锁,肯定不能被表锁的写锁修改.他们的兼容性:

    ISIXSX
    IS兼容兼容兼容不兼容
    IX兼容兼容不兼容不兼容
    S兼容不兼容不兼容不兼容
    X不兼容不兼容不兼容不兼容

5.当前读,快照读

当前读
像select lock in share mode(共享锁), select for update ; update, insert ,delete(排他锁)这些操作都是一种当前读，为什么叫当前读?就是它读取的是记录的最新版本,读取时还要保证其他并发事务不能修改当前记录,会对读取的记录进行加锁.

快照读
像不加锁的select操作就是快照读，即不加锁的非阻塞读；快照读的前提是隔离级别不是串行级别，串行级别下的快照读会退化成当前读；之所以出现快照读的情况，是基于提高并发性能的考虑，快照读的实现是基于多版本并发控制，即MVCC,可以认为MVCC是行锁的一个变种，但它在很多情况下，避免了加锁操作，降低了开销；既然是基于多版本，即快照读可能读到的并不一定是数据的最新版本，而有可能是之前的历史版本

MVCC 中读操作实现的是乐观锁,可以就是读取的是快照.

为了解决’读写’ 冲突,它的实现原理最主要依赖的就是

  • 隐式字段
  • undo log 日志文件
  • ReadView

6.隐式字段

Innodb 存储引擎为数据库表中每行记录都实现了隐藏字段,也就是除了自己定义的字段之外的不可见字段.

  • DB_TRX_ID: 最近修改的事务ID,记录创建这条记录或者是最后一次修改该记录的事务ID
  • DB_ROLL_PTR:回滚指针,指向这条记录的上一个版本(指向当前记录项的rollback segment的undo log记录)
  • DB_ROW_ID:隐藏的主键ID,如果表没有定义主键,InnoDB 会自动在DB_ROW_ID产生一个聚簇索引
  • DELETE BIT:用于标识该记录是否被删除,这里的不是真正的删除数据,真正删除要在commit 执行后.

7.undo log

undo log 分为两种:

  • insert undo log:代表事务在insert 新纪录时产生的undo log ,只在事务回滚时需要,并且在事务提交后可以被立即丢弃
  • update undo log:事务在进行update或delete时产生的undo log ,不仅在事务回滚时需要,在快照读时也需要.所以不能随便删除,只有在快速读或事务回滚不涉及该日志时,对应的日志才会被purge线程统一清除

purge

  • 为了节省磁盘空间,InnoDB有专门的purge线程来清理deleted_bit为true的记录,为了不影响MVCC的正常工作,purge线程自己也维护了一个read view(这个read view相当于系统中最老活跃事务的read view),如果某个记录的deleted_bit为true,并且DB_TRX_ID相对于purge线程的read view可见,那么这条记录一定是可以被安全清除的.

8.事务链表

mysql 中的事务从开始到提交的过程中,都会被保存在一个叫trx_sys 的链表结构中.(trx4是事务ID)

事务链表中保存的都是还没有提交的事务,事务一旦被提交,就会从事务链表中删除.

9.ReadVIew

ReadView 是事务进行乐观锁 的时操作时产生的读视图(view),在执行事务的时候被创建.它包含3个主要的成员:ReadView{ low_trx_id, up_trx_id, trx_ids} ,在并发的情况下,一个事务在启动时,在trx_sys 中保存了所有的未提交事务,那么哪些改变是对当前事务是可见的,哪些是不可见的.这个是通过这个视图ReadVIew 来判断的.

  • low_trx_id:表示该事务启动时,当前事务链表中最大的事务ID,也就是最近创建的除自身之外的最大的事务ID
  • up_trx_id:表示该事务启动时,当前事务链表中最小的事务ID,也就是当前系统中创建最早的还未提交的事务ID.
  • trx_ids:表示所有事务链表中事务的ID集合

transaction id: InnoDB 存储引擎为每一个事务生成的一个唯一ID,也就是表INNODB_TRX 的trx_id 字段的值.这个ID的值会严格按照申请顺序递增.

每次事务更新数据的时候,都会生成一个新的数据版本,并且把transaction id 赋值给这个数据版本的事务ID,记为row trx_id .同时旧的数据版本要保留,并且在数据版本中,能够有信息可以直接拿到它.这就说明了,数据表中的一行记录,可以存在多个版本了的row ,每个版本都有自己的trx_id.这个记录被写在了表中每行的隐藏字段db_trx_id

T2ReadVIew{99,100,}
T3ReadVIew{99,100,101}
T4ReadVIew{99,100,101,102}
T5update from transaction_test
set k=k+1
where id=1
trx_id=102
T6update from transaction_test
set k=k+1
where id=1
trx_id=101
T7select * from
transcation_test
where id=1

随着时间的推进.第一个有效更新是事务C,把数据从(1,1) 变成了(1,2),这时候,这个数据的db_trx_id 是102,原来的90变成了历史版本.

第二个有效更新的是事务B,把数据从(1,2) 变成了(1,3) ,这时候数据的db_trx_id是101,原来的102变成了历史版本

当事务A查询的时候,事务B还没有提交,但是它生成的(1,3) 这个已经是当前版本了.这个数据对象必须是不可读的.否则就是脏读了.

事务A的查询为(事务A的ReadVIew={99,100}):

  • 找到了(1,3)的db_trx_id 是101,属于将开始事务,数据不可见.

  • 根据(1,3) 的DB_ROLL_PTR 和undo log 的对应,回滚找到上一个历史版本(1,2) ,它的db_trx_id 为102,属于将开始事务,数据不可见.

  • 根据(1,2) 的DB_ROLL_PTR 和undo log 的对应,回滚找到上一个历史版本(1,1) ,它的db_trx_id 为90,属于已经提交的事务,数据可见.

这样保证了数据的一致性读.
总结一下,对于一个事务视图来说,除了直接的更新总是可见外,

  • 对于数据版本没有提交的,不可见
  • 对于数据已经提交,但是是在视图创建后提交的.不可见
  • 对于数据已经提交,但是是在视图创建前提交的.可见

回顾以上Excel 中的查询

事务2在T7 时间查看到k 的值是3,而根据一致性读的规则,结果应该是不对的.

  • 事务2的视图数组是先创建的,之后事务3才提交,应该看见(1,2)怎么能算出(1,3)来.

确实如此,如果事务2在更新之前查询一次数据,这个查询返回的k的值确实是1.

但是,当它要去更新数据的时候,就不能再在历史版本上更新了,否则事务3的更新就丢失了.

因此,事务2此时的set k=k+1是在(1,2)的基础上进行的操作,这里就用到了这样一条规则:更新数据都是先读后写的,而这个读,只能读当前的值,称为当前读current read.

当前读:像select lock in share mode(共享锁), select for update ; update, insert ,delete(排他锁)这些操作都是一种当前读，为什么叫当前读?就是它读取的是记录的最新版本,读取时还要保证其他并发事务不能修改当前记录,会对读取的记录进行加锁.

所以,事务2在执行update前,读到的记录的最新版本是(1,2) ,trx_id 是102,update 后,trx_id变成了101,数据变成了(1,3).再执行select 语句的时候,row_id 指向了自己,按照一致性读的规则.自己对自己是可见的.所以就可以读出(1,3).
